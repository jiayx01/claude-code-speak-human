# Before / After

下面是一个典型场景：业务想在房源列表加“急售”标签，希望提升点击率。

## 用户问题

```text
业务要在房源列表加一个“急售”标签，说能提升点击率。
请帮我准备需求评审时该问的问题和判断标准。
```

## Before：容易出现的问题

没有 output style 时，回答通常也能覆盖要点，但容易有这些问题：

- 结构偏模板
- 语言有点硬
- 结论不够像评审现场能直接说的话
- 容易把“点击率提升”当成单点目标，而不是放进用户信任和后链路转化里看

示例：

```text
核心判断：先别急着评估“能不能做”，要先逼问“急售”的定义口径和数据真实性——这类标签最大风险是供给造假和用户信任反噬，而非技术实现。

1. 用户价值
- “急售”对用户意味着什么？是更低价、可议价，还是更快成交？
- 提升点击率是手段还是目的？点进去发现名不副实，反而拉低详情页转化和平台信任。

...
```

这已经不差，但还是偏“标准答案”。

## After：启用 PM Chinese

```text
核心判断：先别急着评估“提升点击率”，要先确认“急售”是真实供给信号还是运营噱头。前者值得做，后者会透支列表页信任。

评审要问清五件事：

1. 用户价值与定义口径
“急售”对买家意味着什么？是降价空间、可议价、可快速成交，还是仅业主主观意愿？没有明确口径，标签就是噪音。判断标准：能写出可验证的客观条件，如降价幅度、挂牌天数、业主成交时限。

2. 供给质量与真实性
哪些房源能打标？谁来认定、数据从哪来？防止“全是急售”导致标签贬值。

3. 滥用与信任风险
经纪人有动机滥用标签骗点击。要问：是否人工可改？如何监控滥用？误标后果谁承担？

4. 效果验证设计
不能只看点击率。主指标可以看 CTR，护栏要看后续咨询率、带看转化、用户投诉。

5. 体验与扩展性
标签会不会挤占其他信息？后续会不会泛滥成“标签墙”？

一句话：要的是“点了之后更可能成交”，不是“更多人点了但都失望”。
```

## 改善点

- 结论更像人话
- 业务判断更明确
- 不是只看点击率，而是把点击率和用户信任、下游转化连起来
- 有结构，但没有模板感太重
