# Before / After

下面是一个更通用的场景：把复杂技术改动讲给非技术同事听。

## 用户问题

```text
研发说要把实时处理链路从同步调用改成异步消息队列，可能会降低接口耗时，但会带来一致性问题。
请帮我转成非技术同事听得懂的版本。
```

## Before：容易出现的问题

没有 output style 时，回答通常也能覆盖要点，但容易有这些问题：

- 概念解释偏多
- 像技术方案摘要，不像给人沟通的话
- 结论不够靠前
- 收益、代价和决策点没有自然连起来

示例：

```text
同步调用是指请求发起后需要等待依赖服务返回结果，异步消息队列则是将任务投递到队列中，由消费者后续处理。这种方式可以降低主链路耗时，提高系统吞吐，但可能导致短时间内数据状态不一致。

需要关注接口耗时、消息堆积、消费失败、重试机制和幂等设计。
```

这不算错，但听起来更像技术笔记。

## After：启用 Human Chinese

```text
核心判断：这个改动会让用户等待时间变短，但也意味着系统里有些结果不会“立刻同步”，需要业务接受短暂延迟。

可以这样和非技术同事解释：

现在的方式像“用户提交请求后，系统必须当场把所有事情办完再回复”，所以稳，但慢。改成消息队列后，会变成“先把关键结果返回给用户，剩下的事情排队处理”，所以页面会更快。

代价是：某些状态可能会晚几秒更新。比如用户刚完成一个动作，后台统计、通知或列表状态可能不是马上变化，而是稍后同步。

我们需要一起确认一个决策点：哪些信息必须实时准确，哪些信息可以接受几秒延迟。前者保留同步，后者适合异步。
```

## 改善点

- 第一段直接给结论
- 解释更像人话，不是术语翻译
- 收益、代价、决策点自然连起来
- 没有牺牲技术准确性
