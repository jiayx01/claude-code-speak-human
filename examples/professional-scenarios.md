# 专业场景示例

这个 output style 适合很多中文工作场景，不只适合产品经理。

## 1. 技术解释

```text
研发说要把实时处理链路从同步调用改成异步消息队列。
请帮我转成非技术同事听得懂的版本。
```

理想回答：

- 少用技术术语
- 讲清收益、代价、影响范围和决策点
- 不牺牲技术准确性

## 2. 方案判断

```text
我们有两个方案：一个上线快但后期维护成本高，一个上线慢但更稳定。
请帮我做一个清楚的决策分析。
```

理想回答：

- 先给判断框架
- 明确短期收益和长期成本
- 说明在什么条件下选 A，什么条件下选 B

## 3. 问题排查

```text
昨天一个核心功能的数据突然下降了 18%，但总访问量基本没变。
请给我一个 30 分钟内能推进的排查路径。
```

理想回答：

- 先给初判
- 先排数据/系统问题，再拆用户、渠道、版本、功能变化
- 明确 30 分钟内能定位到哪一层，不承诺一步找到根因

## 4. 文案改写

```text
请把这段话改得更清楚、更像人说的话：
由于系统架构升级，部分数据同步链路将进行异步化改造，相关指标可能产生短时间延迟。
```

理想回答：

- 直接给可用版本
- 少说术语
- 保留必要风险，不把问题说没了

## 5. 代码解释

```text
请解释这段脚本的核心逻辑，不要逐行翻译。
重点告诉我它解决什么问题、输入输出是什么、哪里可能出错。
```

理想回答：

- 先说脚本目的
- 再说关键流程
- 最后说风险点和验证方式
