# 评分标准

每项 1-5 分，5 分最好。

| 维度 | 5 分标准 | 常见扣分 |
| --- | --- | --- |
| 中文自然度 | 像中文 PM 直接写出来，顺畅、克制、少翻译腔 | 生硬、套话多、英文结构明显 |
| 结论清晰度 | 开头有明确判断，能指导下一步 | 先铺背景，结论藏在后面 |
| 逻辑连贯性 | 因果、递进、取舍关系清楚 | 跳跃、并列堆点、前后不呼应 |
| 层次结构 | 结构服务理解，分层不过度 | 标题过多或一整段糊在一起 |
| 产品经理可读性 | 能连接业务目标、数据口径和产品动作 | 技术/概念表达过重 |
| 技术准确性 | 不牺牲事实、口径和技术约束 | 顺但虚、看似合理但失真 |
| 信息密度 | 每段都有有效信息 | 正确废话多、泛泛而谈 |
| 不确定性表达 | 能区分事实、推断、假设、待验证项 | 把猜测写成确定事实 |
| 可执行性 | 有明确动作、检查项或决策规则 | 停留在原则建议 |
| 简洁度 | 不缺关键点，也不明显啰嗦 | 冗长、重复、过度解释 |

## 总分解释

- 45-50：可直接作为目标风格。
- 40-44：基本可用，有局部坏味道。
- 35-39：信息可用，但表达或结构需要修。
- 30-34：读起来费劲，只有部分内容可借鉴。
- 30 以下：不适合作为最终配置效果。

## 避免偏差

- 不因答案更长而加分；长答案必须有更高的信息密度。
- 不因语气更顺而忽略事实失真。
- 不因结构复杂而默认更专业。
- 对产品问题，优先看是否能帮助用户做决策。
